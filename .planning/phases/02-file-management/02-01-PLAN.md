# Plan 02-01: File Management API Enhancement

**Phase:** 2 - File & Project Management
**Plan:** 1 of 4
**Type:** Feature
**Wave:** 1 (Can start immediately)
**Depends on:** Nothing (uses existing backend code)
**Files modified:** `backend/file_service.py`, `backend/file_api.py`
**Autonomous:** Yes (can execute independently)
**Must haves:**
1. All file types from requirements supported (PDF, DOCX, MD, .py, .r, .js, CSV, Excel)
2. File size validation and limits enforced
3. Duplicate file names handled with auto-renaming
4. File metadata extraction (type detection, size, hash)
5. Error handling for invalid files

## Overview

Enhance existing file management backend to support all required file types from requirements, add proper validation, handle duplicates, and prepare for cloud storage migration. The backend code exists but needs hardening for production use.

## Tasks

### Task 1: Add comprehensive file type validation and mime type detection

**File:** `backend/file_service.py`

**Changes:**
- Add file type whitelist based on requirements (PDF, DOCX, MD, .py, .r, .js, CSV, Excel)
- Use `python-magic` or `mimetype` library for robust mime type detection
- Map file extensions to mime types for all supported types
- Add file size validation (configurable max size, default 50MB)
- Return clear error messages for unsupported types

**Success criteria:**
- Upload rejected for unsupported file types with clear error message
- Upload rejected for files exceeding size limit
- All 8 required file types accepted: `.pdf`, `.docx`, `.md`, `.py`, `.r`, `.js`, `.csv`, `.xlsx`

### Task 2: Implement duplicate file name handling

**File:** `backend/file_service.py` (upload_file function)

**Changes:**
- Check for existing files with same name in target folder
- Auto-rename duplicates with pattern: `filename (1).ext`, `filename (2).ext`
- Update path generation to handle renamed files
- Ensure storage paths remain unique

**Success criteria:**
- Uploading `paper.pdf` twice creates `paper.pdf` and `paper (1).pdf`
- No database constraint violations on duplicate names
- File tree displays both files correctly

### Task 3: Add file metadata extraction

**File:** `backend/file_service.py`

**Changes:**
- Extract metadata for specific file types:
  - PDF: page count, title, author (using PyPDF2 or pdfplumber)
  - CSV: row count, column names
  - Excel: sheet names, row counts
  - Code files (.py, .r, .js): line count
- Store extracted metadata in File model's JSONB field
- Add metadata to API response

**Success criteria:**
- PDF metadata (page count, title) extracted and stored
- CSV/Excel sheet info extracted
- Code file line counts available
- Metadata visible in API response

### Task 4: Improve error handling and logging

**Files:** `backend/file_service.py`, `backend/file_api.py`

**Changes:**
- Add specific exception types (FileTooLargeError, UnsupportedFileTypeError)
- Log all file operations with details (user, project, file, size)
- Return structured error responses to frontend
- Add transaction safety for DB operations

**Success criteria:**
- All errors logged with context
- Frontend receives actionable error messages
- Failed uploads don't create partial records
- Database transactions rolled back on errors

## Definition of Done

- [ ] All 8 required file types upload successfully
- [ ] File size limits enforced (configurable via env var)
- [ ] Duplicate filenames auto-renamed
- [ ] Metadata extracted for PDF, CSV, Excel, code files
- [ ] Error handling comprehensive with structured responses
- [ ] All operations logged
- [ ] API documentation updated (OpenAPI specs)
- [ ] Tested with sample files of each type

## Testing Strategy

1. Upload each of the 8 supported file types
2. Upload unsupported file type → verify rejection
3. Upload file exceeding size limit → verify rejection
4. Upload same file twice → verify auto-renaming
5. Check metadata extraction for PDF (page count), CSV (rows), code (lines)
6. Verify error messages are clear and actionable
