<task type="auto">
  <name>Task 1: Create ExecutionService for sandboxed code execution</name>
  <files>backend/execution_service.py</files>
  <action>
Create backend/execution_service.py:

1. ExecutionService class with methods:
   - execute_python(code: str, timeout: int = 60) -> ExecutionResult
   - execute_r(code: str, timeout: int = 60) -> ExecutionResult
   - _execute_subprocess(command: list, code: str, timeout: int) -> ExecutionResult

2. ExecutionResult Pydantic model:
   - success: boolean
   - output: string (stdout)
   - error: string (stderr or exception message)
   - execution_time: float (seconds)
   - return_code: int

3. Python execution (execute_python):
   - Use subprocess.run() with isolated environment
   - Command: ["python3", "-c", code]
   - Capture stdout and stderr separately
   - Set timeout parameter (default 60 seconds)
   - Restrict built-ins for safety: Remove __import__, open, eval, exec from globals
   - Working directory: /tmp or dedicated workspace directory

4. R execution (execute_r):
   - Use subprocess.run() with Rscript command
   - Command: ["Rscript", "-e", code]
   - Capture stdout and stderr
   - Set timeout parameter (default 60 seconds)
   - Working directory: /tmp or dedicated workspace directory

5. Error handling:
   - Catch TimeoutExpired and return error message
   - Catch subprocess.SubprocessError and return error
   - Validate code length (max 10000 characters) to prevent abuse

6. Output capture:
   - Decode stdout/stderr as UTF-8 with error handling
   - Trim excessive output (max 100000 characters)
   - Include execution time in result

Use subprocess module with proper timeout and capture_output=True.
No external dependencies (Docker, containers) for MVP simplicity.
</action>

<task type="auto">
  <name>Task 2: Create code execution API endpoint</name>
  <files>backend/analysis_api.py</files>
  <action>
Update backend/analysis_api.py with execution endpoint:

1. POST /api/analysis/projects/{project_id}/execute
   - Accepts: {code: string, language: "python" | "r", save_to_memory: boolean}
   - Validates code length and language parameter
   - Calls ExecutionService.execute_python() or execute_r()
   - If save_to_memory=true, calls MemoryService.create_finding() with results
   - Returns: {success: boolean, output: string, error: string, execution_time: float, finding_id: int | null}

2. Request model (Pydantic):
   - code: str (min_length 1, max_length 10000)
   - language: Literal["python", "r"]
   - save_to_memory: bool (default true)

3. Response model (Pydantic):
   - success: bool
   - output: str
   - error: str
   - execution_time: float
   - finding_id: Optional[int]

4. Error handling:
   - Return 400 for invalid language or empty code
   - Return 413 for code exceeding max length
   - Return 500 for execution service failures
   - Include detailed error messages in response

5. Integration with MemoryService:
   - If save_to_memory=true and execution succeeded:
     - Create Finding with finding_type="analysis"
     - Store output in finding_data JSONB field
     - Set source_type="ANALYSIS", source_id=execution timestamp
     - Include code, language, execution_time in metadata

Follow existing API patterns from chat_api.py and memory_api.py.
</action>
